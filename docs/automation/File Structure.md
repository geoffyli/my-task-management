# File Structure

This document describes every file type in the `automation/` directory and whether it is written by hand or generated by the Windmill CLI.

## Top-Level Files

| File | Purpose |
|---|---|
| `automation/wmill.yaml` | Windmill CLI sync configuration. Scoped to `f/notion_tasks/**` and `f/notion/**`. |
| `automation/wmill-lock.yaml` | Lock state for sync operations. Auto-generated; do not edit manually. |
| `automation/rt.d.ts` | TypeScript resource type namespace definitions. Generated by `wmill resource-type generate-namespace`. |

## The f/ Directory

All Windmill entities live under `automation/f/`. The path convention is:

```
automation/f/<folder_name>/<entity_name>
```

Current structure:

```
automation/f/
  notion_tasks/
    create_repetitive_tasks.ts
    create_repetitive_tasks.script.yaml
    create_repetitive_tasks.script.lock
    create_repetitive_tasks_schedule.schedule.yaml
    create_weekly_note.ts
    create_weekly_note.script.yaml
    create_weekly_note.script.lock
    create_weekly_note_schedule.schedule.yaml
    update_legacy_tasks.ts
    update_legacy_tasks.script.yaml
    update_legacy_tasks.script.lock
    update_legacy_tasks_schedule.schedule.yaml
  notion/
    folder.meta.yaml
```

## Entity File Patterns

| Pattern | Entity Type | Notes |
|---|---|---|
| `*.ts` | Script source code | The `main` function is the entry point. |
| `*.script.yaml` | Script metadata | Summary, description, schema. Generated by `wmill script generate-metadata`. |
| `*.script.lock` | Dependency lockfile | Pinned dependency versions. Auto-generated alongside `.script.yaml`. |
| `*.schedule.yaml` | Schedule | Cron-based trigger configuration. Uses 6-field cron (includes seconds). |
| `*.http_trigger.yaml` | HTTP trigger | Custom HTTP endpoint configuration. |
| `folder.meta.yaml` | Folder metadata | Folder summary and permissions. |

## Generated vs Hand-Written

| File Type | Written By | When to Regenerate |
|---|---|---|
| Script source (`.ts`) | Developer | -- |
| `.script.yaml` | CLI (`generate-metadata`) | After changing the function signature or imports. |
| `.script.lock` | CLI (`generate-metadata`) | After changing imports. |
| `.schedule.yaml` | Developer | -- |
| `.http_trigger.yaml` | Developer | -- |
| `rt.d.ts` | CLI (`resource-type generate-namespace`) | After adding or modifying custom resource types. |
| `wmill-lock.yaml` | CLI (`sync`) | Automatic; regenerated on every sync operation. |

## Typical Regeneration Workflow

After editing a script's function signature or adding a new import:

```bash
cd automation

# Regenerate metadata for a single script
wmill script generate-metadata f/notion_tasks/create_repetitive_tasks.ts --yes

# Or regenerate metadata for all scripts at once
wmill generate-metadata --yes
```
